## Problem Statement

中国中型以上企业法务部门在管理多项目、多主体时，面临严重的信息碎片化问题：

- 项目管理靠人工维护表格，与 OA 审批、历史合同、风险记录完全脱节
- 合同审查时需要人工翻找海量历史合同和风险备忘录，耗时且依赖个人经验
- 历史法律文件（合同、风险备忘录）无法直接作为 AI 判断的上下文
- 证照/项目信息系统只能提醒到期，无法判断优先级，无法自动发起维护

核心断点：**AI 工具没有上下文，法务团队的知识积累对 AI 透明**。

## Demand Evidence

- 第一手需求：创始人本人在企业法务部门工作，每天经历上述问题
- 普遍性判断：接触过的同行法务团队有类似状况，问题不是个案
- 现有"解法"：多套互不相通的系统（OA + Excel + CLM + IM），人工复制粘贴是唯一集成方式

## Status Quo（现状成本）

典型企业法务的拼凑方案：
- Excel/飞书表格（项目台账）→ OA 系统（合同审批）→ 邮件/IM（沟通）→ CLM 系统（合同存档）
- 每套系统的数据对其他系统不可见
- 所有"上下文联动"靠人工搜索、复制、粘贴
- 新法务成员上手需要 3-6 个月的经验积累才能建立"人脑索引"

## Target User & Narrowest Wedge

**目标用户（最痛的那个人）：**
企业内部法务团队成员（in-house legal counsel），需要在合同审查或项目判断时快速调取相关历史上下文的人。他们已经在用 AI（Claude/Deepseek/Qwen 等），但 AI 没有公司数据，每次都要手动喂上下文。

**最小切入点（Narrowest Wedge）：**
**项目信息查询** — 不是合同审核，而是让 AI 能直接回答"这个项目现在在哪个阶段？关联了哪些合同、证照、风险记录？"

路径：
1. 一个轻量数据录入界面，供法务团队录入/维护项目基本信息（项目名、阶段、关联合同、证照、风险条目）
2. MCP Server 将这些结构化数据暴露给 AI 客户端
3. 法务成员在 Claude Desktop / Cursor 中直接问"XX 项目现状？"，AI 能给出有依据的答案

不需要解析历史文件，不需要 OCR，不需要替换任何现有系统。数据从录入界面进来，AI 从 MCP 工具读出去。

## Constraints

- 中国企业数据安全要求：企业数据不出内网，MCP 服务器必须支持私有化部署
- 目标用户的 AI 使用习惯：法务团队已在用 Claude/Deepseek，不愿意再学一套界面
- 冷启动：第一个客户就是创始人自己的公司，可以快速迭代

## Premises

1. **碎片化上下文是真正的瓶颈** — 不是 AI 能力不够，是 AI 拿不到数据，agree ✅
2. **In-house legal，不是律所** — 完全不同的销售和产品逻辑，agree ✅
3. **MCP 协议层比界面先行更快到达价值** — 最小部署成本，最快看到 AI 效果，agree ✅
4. **竞品都在做"AI 进 CLM"，没有人做"上下文协议层"** — 市场扫描印证，agree ✅

## Approaches Considered

### Approach A: 上下文聚合层（最小切入）
**Summary:** 不替换现有系统，先做连接层——把 OA/表格/邮件/历史合同汇聚成 AI 可读的结构化上下文。
**Effort:** M（人力 3 周 / CC+gstack ~5 天）
**Risk:** 低
**Pros:**
- 最快交付可见价值（3-4 周有可演示版本）
- 不需要法务团队改变工作流
- 验证核心假设：上下文聚合 → AI 判断效率提升
**Cons:**
- 只是数据管道，没有 Agent 原生能力
- 难以建立护城河（竞品可以快速复制）
**Reuses:** 现有 demo 的设计语言，v2.2 上下文引擎设计原则

### Approach B: Agent 原生法务工作台（完整愿景）
**Summary:** 按 v2.2 设计文档全栈实现：项目/主体双根上下文、任务载体、确认中心、工具调用。
**Effort:** XL（人力 6+ 个月 / CC+gstack 3-4 周）
**Risk:** 高（路程长，第一个客户等待时间过长）
**Pros:**
- 完整占领法务 Agent 工作台市场
- 护城河深（数据 + 工作流 + 规范包）
**Cons:**
- 第一个可付费版本需要 3 个月以上
- 过于复杂，容易陷入过度设计
**Reuses:** v2.2 设计文档全部模块

### Approach C: MCP 法务上下文服务器（推荐）⭐
**Summary:** 不做 UI，先做协议层——将企业法务数据暴露为 MCP Server，让任意 AI 客户端（Claude Desktop、Cursor、Deepseek 等）直接调用公司内部法律上下文。
**Effort:** M（人力 4 周 / CC+gstack ~2 周开发 + 第 1 周用户观察）
**Risk:** 中（客户需要具备 AI 使用能力，需要企业允许内网部署）
**Pros:**
- 直接对接法务团队已有的 AI 使用习惯，0 新工作流学习成本
- 协议中立，不绑定任何 AI 模型
- 私有化部署，天然解决数据安全合规
- 是 v2.2 MCP-First 长期架构的第一层实现
- 建立"数据是护城河"的竞争优势
**Cons:**
- 需要客户有 AI 客户端使用能力（目前早期市场）
- 货币化模式需要探索（MCP Server 怎么卖？）
- 无产品界面，难以做销售演示
**Reuses:** v2.2 设计文档的上下文种子模式、rationale 强制入参设计

## v1 产品结构（两件事，缺一不可）

**件 1：数据录入界面（Web UI）**
轻量的内网 Web 应用，供法务团队录入和维护项目结构化信息：
- 新建/编辑项目：名称、阶段（谈判中/签约/执行/关闭）、备注
- 关联合同：合同名称、对方主体、签署日期、摘要（手动填写，v2 再做文件解析）
- 关联证照：证照类型、归属主体、到期日
- 关联风险记录：风险描述、状态（待处理/已关闭）、来源备注
- 录入界面不要求美观，只要快速录入不丢信息

**件 2：MCP Server**
将录入的数据暴露为 AI 可调用工具：

**错误处理统一约定：** 资源不存在返回 `null`（列表类返回 `[]`），服务故障返回 `{ error: string }`。

```
list_projects(stage?: string)
  → [{ project_id, name, stage, contract_count, license_count, open_risk_count, last_updated }]
  → 给 AI 提供项目总览，可按阶段筛选

get_project_context(project_id_or_name: string)
  → { project_id, name, stage, notes,
      contracts: [{ title, counterparty, signed_date, summary }],
      licenses: [{ type, entity, expiry_date, days_until_expiry }],
      risks: [{ description, status, source }],
      last_updated }
  → 核心工具：回答"XX 项目现在什么情况"
  → 支持项目名模糊匹配（jieba + 编辑距离 ≥ 70）

list_expiring_licenses(days_ahead: int=30)
  → [{ license_type, entity, expiry_date, project_name, priority_score }]
  → priority_score = max(0, 100 - days_until_expiry)（v1 纯时间函数）

list_open_risks(project_id?: string)
  → [{ project_name, description, status, created_date, source }]
  → 不传 project_id 时返回全部未关闭风险
```

**数据模型（v1，SQLite）：**
- `projects`: id, name, stage, notes, created_at, updated_at
- `contracts`: id, project_id, title, counterparty, signed_date, summary
- `licenses`: id, project_id, entity_name, license_type, expiry_date
- `risks`: id, project_id, description, status(open/closed), source, created_at

**Auth 模型（v1）：** 本地 API Key（内网单租户，不暴露外网）。录入界面无鉴权（内网访问即可），v2 再加角色权限。

## Recommended Approach

**选择 Approach C（MCP 法务上下文服务器）**，原因：

v1 的核心价值主张：法务成员在 Claude Desktop 里问"XX 项目现在什么情况"，AI 能直接给出有依据的项目阶段、关联合同状态、证照风险汇总——而不是靠人脑记忆或翻多套系统。

**实施时序：**
- 第 0 周（已清除）：IT 内网部署已确认，IP 归属走内部创业路径 ✅
- 第 1 周：录入界面（Web UI）+ 基础数据模型开发
- 第 2 周：MCP Server 开发 + 与 Claude Desktop 联调
- 第 3 周：录入真实项目数据，本人日常使用，找 1 个同事试用
- 第 4 周：根据真实使用反馈迭代，确认成功信号

成功信号（4 周后应达到）：
- 使用 Claude Desktop 审查合同时可调用公司历史合同上下文，减少翻档时间
- 至少 1 个同事每周主动使用超过 3 次
- 能量化描述"没有这个工具，我需要多花 X 分钟"

**货币化假设（待验证）：** 按部署企业收取年费（建议初期定价 2-5 万/年/企业），核心卖点是私有化部署 + 数据安全合规。这是假设，需在第一个付费客户时验证。

## Engineering Review Decisions

Generated by `/plan-eng-review` on 2026-05-21.

Implementation source of truth: `Docs/v1-implementation-plan.md`.

### v1 Scope

v1 is reduced from "Web UI + MCP Server + Docker deployment" to a smaller proof:

1. Ship a local stdio MCP server.
2. Keep the server client-agnostic, with setup support for Claude Desktop, Cursor, Codex, and generic stdio MCP clients.
3. Use SQLite as the canonical local database.
4. Use CSV/XLSX import as the data entry path.
5. Defer Web GUI, Docker/HTTP deployment, OCR/PDF extraction, OA/Feishu/CLM connectors, and enterprise permission UI.

### Data Flow

```text
Raw CSV/XLSX from user
      |
      v
AI or template normalization
      |
      v
legal-mcp import
      |
      v
shared import pipeline
  parse -> normalize -> validate -> upsert
      |
      v
SQLite canonical DB
      |
      v
stdio MCP server
      |
      v
Claude / Cursor / Codex / generic MCP client
```

### Implementation Requirements

1. `project_code` is the stable unique project identifier. Project names are display fields only and may duplicate or change.
2. Child records use stable external keys unique within each project.
3. Imports are idempotent upserts and produce row-level created/updated/skipped/failed reports.
4. Import validation lives in a shared pipeline used by CLI import now and future GUI upload later.
5. Every MCP tool requires `rationale` and logs timestamp, tool name, arguments summary, source client when available, and stable error code on failure.
6. MCP tool errors use structured objects with stable `error.code`, not free-text strings.
7. Project lookup is deterministic: exact `project_code` first, exact unambiguous name second, fuzzy only when a single candidate clears threshold. Ambiguity returns candidates instead of guessing.
8. SQLite schema includes unique constraints and indexes for project code, stage, name, child external keys, license expiry, and risk status.
9. v1 distribution uses a one-line installer that bootstraps `uv tool install legal-mcp`, then enters `legal-mcp setup`.
10. `legal-mcp setup` lets the user choose AI desktop clients and writes the needed MCP configuration.
11. `legal-mcp doctor` verifies install, database, tool discovery, and sample query health.
